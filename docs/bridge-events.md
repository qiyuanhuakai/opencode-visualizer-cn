# Bridge Events

This document describes the SSE event stream shape used by `vis-bridge`.

The goal is similar to [SSE.md](./SSE.md): this file alone should be enough to implement a parser and understand event ordering at the bridge layer.

---

## Endpoint

### `GET /bridge/events`

Streams normalized events from the bridge.

Each SSE `data:` line contains a JSON object:

```text
data: {"id":"evt_123","type":"message.delta","time":1760000000000,"sessionID":"ses_123","providerID":"codex","data":{...}}
```

The bridge does not need SSE `event:`, `id:`, or `retry:` fields for the first version.

### Event Envelope

- `id` (string) — unique event ID generated by the bridge
- `type` (string) — normalized event name
- `time` (number) — Unix timestamp in milliseconds
- `sessionID?` (string) — present for session-bound events
- `providerID?` (string) — present when the event is associated with a provider
- `pluginID?` (string) — present when the event is associated with a plugin
- `data` (object) — event-specific payload

### Ordering Guarantees

The bridge should guarantee ordering **within a single session**.

Across different sessions, consumers should treat interleaving as best effort and must not assume global ordering by arrival time.

## Event Types

### Bridge Lifecycle

- `bridge.ready`
  - `version: string`
  - `startedAt: number`
- `bridge.degraded`
  - `healthy: boolean`
  - `notes: string[]`

### Provider

- `provider.updated`
  - `provider: ProviderInfo`

Emitted when:

- a provider probe result changes
- a provider becomes unavailable
- a provider version or capability set changes

### Plugin

- `plugin.updated`
  - `plugin: PluginInfo`

Emitted when:

- a plugin becomes available/unavailable
- a plugin is lazily activated
- a plugin is disabled or unloaded

### Session

- `session.created`
  - `session: SessionInfo`
- `session.updated`
  - `session: SessionInfo`
- `session.completed`
  - `sessionID: string`
  - `status: "completed" | "cancelled" | "error"`
- `session.error`
  - `sessionID: string`
  - `error: BridgeErrorData`

### Message

- `message.started`
  - `sessionID: string`
  - `messageID: string`
  - `role: "assistant" | "user" | "system"`
- `message.delta`
  - `sessionID: string`
  - `messageID: string`
  - `delta: string`
- `message.completed`
  - `sessionID: string`
  - `messageID: string`
  - `text: string`
  - `usage?`: [UsageInfo](#usageinfo)

`message.delta` is the primary streaming event for assistant output.

### Tool

- `tool.started`
  - `sessionID: string`
  - `toolCallID: string`
  - `toolName: string`
  - `title?`: string
  - `metadata?`: Record<string, unknown>
- `tool.completed`
  - `sessionID: string`
  - `toolCallID: string`
  - `toolName: string`
  - `output?`: unknown
- `tool.failed`
  - `sessionID: string`
  - `toolCallID: string`
  - `toolName: string`
  - `error: BridgeErrorData`

### Approval

- `approval.requested`
  - `sessionID: string`
  - `requestID: string`
  - `kind: "permission" | "tool" | "command"`
  - `title: string`
  - `details?`: string
  - `metadata?`: Record<string, unknown>
- `approval.resolved`
  - `sessionID: string`
  - `requestID: string`
  - `decision: "once" | "always" | "reject"`

### Optional Future Events

The following are intentionally excluded from the first contract, but may be added later if needed:

- `pty.created`
- `pty.updated`
- `pty.deleted`
- `plugin.metric.updated`
- `status.snapshot`

## Shape References

### ProviderInfo

- `id`: string
- `label`: string
- `available`: boolean
- `version?`: string
- `reason?`: string
- `capabilities`: `{ streaming: boolean; resume: boolean; approval: boolean; tools: boolean; structuredOutput: boolean }`

### PluginInfo

- `id`: string
- `label`: string
- `source`: `bridge-config` | `project-config` | `discovery`
- `available`: boolean
- `loaded`: boolean
- `path?`: string
- `reason?`: string
- `contributes`: (`tool` | `command` | `context` | `status` | `metrics`)[]

### SessionInfo

- `id`: string
- `providerID`: string
- `directory?`: string
- `title?`: string
- `model?`: string
- `status`: `idle` | `running` | `awaiting-approval` | `completed` | `error` | `cancelled`
- `createdAt`: number
- `updatedAt`: number
- `pluginIDs`: string[]
- `lastError?`: BridgeErrorData
- `metadata`: Record<string, unknown>

### BridgeErrorData

- `code`: string
- `message`: string
- `providerID?`: string
- `pluginID?`: string
- `sessionID?`: string
- `details?`: Record<string, unknown>

### UsageInfo

- `input?`: number
- `output?`: number
- `reasoning?`: number
- `cacheRead?`: number
- `cacheWrite?`: number
- `contextLimit?`: number

## Parsing Notes

1. Consumers should ignore unknown event types for forward compatibility.
2. `message.delta` events may be very frequent; UI consumers should batch renders when needed.
3. `session.updated` should be treated as authoritative for coarse session state, while `message.*`, `tool.*`, and `approval.*` provide fine-grained progress.
4. The bridge should normalize provider-native interruptions or crashes into `session.error` with a stable `BridgeErrorData.code`.
